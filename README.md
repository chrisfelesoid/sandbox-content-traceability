# sandbox-ai-traceability

Evaluating digital watermarking solutions for content traceability: A unified verification environment (A personal study).


## Use case

All use cases also utilize watermarking and C2PA.
This verification assumes that C2PA has been removed.

**Disclaimer on Robustness:**
* **Digital watermarking solutions** do not provide an absolute guarantee of traceability under all conditions.
* Due to constantly evolving watermark removal technologies and malicious attacks, maintaining a 100% detection rate remains technically and practically impossible.

### 1. Product Provenance Tracking

Trace which specific product, platform, or version the content originated from.
* **Target Data to Embed:** Company/Product identifiers (e.g., Domain Name, Corporate Business Number, etc.)
* **Minimum Bits Required:** **40 bits**

### 2. Content Leakage & Distribution Tracking

Identify the source and distribution channel of leaked or unauthorized content.
* **Target Data to Embed:** Unique Content ID (This use case frequently includes the metadata from *Use Case 1: Product Provenance Tracking* to trace both the product and the specific content).
* **Minimum Bits Required:** **40 bits**

### 3. AI-Generated Content Identification

Distinguish whether a piece of content was generated by AI or created by a human.
* **Target Data to Embed:** AI-generation flag (`0` or `1`), Identifier of the generator/product used for creation, etc.
* **Minimum Bits Required:** **16 bits**


## Summary of Bit Requirements

The table below outlines the minimum payload size required to support each use case based on the specifications:

| Use Case | Description                             | Minimum Bits Required |
|----------|-----------------------------------------|-----------------------|
| **1**    | Product Provenance Tracking             | **40 bits**           |
| **2**    | Content Leakage & Distribution Tracking | **40 bits**           |
| **3**    | AI-Generated Content Identification     | **16 bits**           |

Note:


### 16-Bit Watermark Payload Example Specification

| Bit Range | Size   | Purpose             | Description                                                                                                   |
|-----------|--------|---------------------|---------------------------------------------------------------------------------------------------------------|
| Bit 0     | 1 bit  | AI/Human Flag       | 0: Human-created / 1: AI-generated                                                                            |
| Bit 1–4   | 4 bits | AI Model Type       | Identifies the generator type (e.g., LLM, Diffusion, Audio AI, or specific model IDs. Max 16 types).          |
| Bit 5–9   | 5 bits | Product/Platform ID | Identifies which specific tool, platform, or service generated the content (Max 32 IDs).                      |
| Bit 10–13 | 4 bits | Version             | Control	Tracks the watermark algorithm version or model version for backward compatibility (Max 16 versions). |
| Bit 14–15 | 2 bits | Error Detection     | Parity bits to verify data integrity and detect tampering.                                                    |

#### Technical Limitation of 16-Bit Payload

Please note that a **16-bit payload is strictly insufficient** for tracking unique companies or individual content items.

* **The Math:** A 16-bit space can only represent $2^{16} = 65,536$ unique values.
  * Uses 1 bit for the AI/Human Flag and 2 bits for the Error Detection Checksum, 13 bits can only represent $2^{13} = 8,192$ unique values.
* **Why it fails for Use Cases 1 & 2:**
  * **Company/Product Tracking:** It cannot accommodate globally unique identifiers such as corporate registration numbers or domain names (which require larger bit strings or hashing, typically needing at least 40 bits).
  * **Content Tracking:** It cannot store unique Content IDs or UUIDs for large-scale distribution, as it would cause immediate ID collisions.

Therefore, **16 bits is dedicated exclusively to Use Case 3 (AI Identification & Basic Model Tagging)**. For robust product and leakage tracking, a minimum of **40 bits** is required.


### 40-Bit Watermark Payload Example Specification

| Bit Range | Size    | Purpose             | Description                                                                                |
|-----------|---------|---------------------|--------------------------------------------------------------------------------------------|
| Bit 0-13  | 14 bit  | Product/Platform ID | Identifies which specific tool, platform, or service generated the content (Max 16384 IDs) |
| Bit 14–37 | 24 bits | Content ID          | Identifies the content (Max 16 million IDs).                                               |
| Bit 38–39 | 2 bits  | Error Detection     | Parity bits to verify data integrity and detect tampering.                                 |

#### Technical Limitation of 40-Bit Payload

Please note that although a **40-bit payload remains strictly insufficient**, it is still manageable with proper system design.

* **The Math:** A 16-bit space can only represent $2^{40} \approx 1 trillion$ unique values.
  * Uses 1 bit for the AI/Human Flag and 2 bits for the Error Detection Checksum, 13 bits can only represent $2^{13} = 8,192$ unique values.

The "Content ID" embedded in this watermark does not have the capacity of a standard `UUID` or `BIGINT` (64-bit). Therefore, since these IDs must be reused on a rotational basis, it cannot represent the raw asset ID directly. Instead, it must function as an **intermediary identifier (lookup key)** to resolve the actual metadata managed in an external database, such as C2PA.

### Switching CUDA Versions via Command Line

Since several versions of CUDA drivers are pre-installed within the container, you can easily switch between them using a command. This allows you to select the specific environment required for your project.

To change the active CUDA version, use the following command:

```
sudo update-alternatives --config cuda
```
